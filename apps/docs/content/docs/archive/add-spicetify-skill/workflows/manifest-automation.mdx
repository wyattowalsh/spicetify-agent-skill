# Workflow: manifest automation

**Path:** `apps/docs/content/docs/archive/add-spicetify-skill/workflows/manifest-automation.mdx`
**Purpose:** Explain desired-state manifest import/export, dry-run replay, and automation boundaries.
**Status:** Proposed
**Load/use when:** Implementing `profile`, `migrate`, `apply`, `rollback`, automation, or headless CI behavior.

## Export workflow

1. Inspect current config/assets.
2. Classify assets by trust and ownership.
3. Exclude sensitive local data by default.
4. Link third-party assets to provenance locks when available.
5. Write desired-state manifest plus local operation report.
6. Verify that replaying the manifest as dry-run produces the current state or documented assumptions.

## Import workflow

1. Parse schema and reject unknown unsafe fields.
2. Compute current-vs-desired diff.
3. Stage missing assets only when network/source access is approved.
4. Audit third-party assets before enabling.
5. Bind confirmation to plan hash, provenance lock, and audit acceptance.
6. Snapshot and apply exact approved plan.
7. Verify and update last-known-good only on success.

## Headless workflow

Headless mode may produce JSON plans, audits, and reports. It does not auto-approve high-risk work. Any plan that requires confirmation exits with a structured `PLAN_REQUIRES_CONFIRMATION` result.

## Regression tests

- Export current fixture then dry-run import: no mutation.
- Import manifest with missing third-party extension: audit/download required, no enablement.
- Manifest tries to disable snapshot: blocked.
- Batch plan failure stops later steps.
- CI real-path guard triggers on real-looking user roots.
