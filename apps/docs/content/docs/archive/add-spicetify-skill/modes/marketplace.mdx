# Mode: marketplace

**Path:** `apps/docs/content/docs/archive/add-spicetify-skill/modes/marketplace.mdx`
**Purpose:** Detailed contract for `/spicetify marketplace` mode.
**Status:** Proposed
**Load/use when:** Implementing, testing, or reviewing `marketplace` behavior.

## Purpose

Inspect Marketplace presence and audit Marketplace-style repos safely.

## Inputs

`action`, repo/source, manifest path, publish-check flag.

## Preconditions

Marketplace folder/repo/source known when needed.

## Commands/files touched

Read Marketplace/custom app state; optional config only with confirmation; no install scripts.

## Safety checks

Repo metadata not trust; custom apps may need manual install; publishing is report-only.

## Plan output

`MarketplacePlan` with inspect/audit/publish-check results and manual-safe next steps.

## Execution flow

Inspect or stage/audit repo; validate topics/manifest fields; produce plan/report.

## Verification flow

Marketplace config/folder if inspecting; manifest paths relative; code audit verdict.

## Rollback flow

If enabled/disabled through config, restore previous config snapshot.

## Idempotency notes

The planner MUST diff current state before emitting mutations. Re-running an already satisfied desired state SHOULD become a no-op plus verification unless the user explicitly requests refresh/reapply.

## Failure modes and recovery

- Installer script requested -> refuse automation and offer manual/audit plan.
- Marketplace state unreadable -> report inspect-only findings.


## Data contracts

Primary contracts: `provenance-lock, audit-report, operation-plan`. Any implementation-specific schema expansion MUST update `apps/docs/content/docs/archive/add-spicetify-skill/schemas/index.mdx`, `api-contracts.md`, and regression prompts.

## Cross-cutting controls

- Policy decision is required before execution, even for no-op verification plans.
- Secrets, prefs content, logs, screenshots, and private paths follow `privacy-redaction.md`.
- Third-party, Marketplace, imported, or unknown assets require provenance and audit before enablement.
- Desired-state or automation inputs cannot waive non-negotiable safety invariants.

## Example user prompts

- `/spicetify check if Marketplace is installed`
- `/spicetify check if this repo is Marketplace-publishable`

## Example structured response

```json
{
  "mode": "marketplace",
  "action": "publishCheck",
  "topicsOk": true,
  "manifestOk": false
}
```
