# Docs site implementation plan

**Path:** `apps/docs/content/docs/archive/add-spicetify-skill/docs-site-implementation-plan.mdx`
**Purpose:** Dependency-aware implementation sequence for the Fumadocs + shadcn/ui site.
**Status:** Proposed
**Load/use when:** Handing the docs-site work to Codex or slicing it into implementation tasks.

## Implementation waves

### W0 — Preflight

- Confirm target repo package manager and workspace conventions.
- Confirm whether the site lives in `apps/docs` or a repo-specific docs path.
- Recheck current Fumadocs and shadcn/ui setup docs.
- Record decisions in `apps/docs/content/docs/archive/add-spicetify-skill/context-map.mdx` and `decision-records.md`.

### W1 — App shell and content source

- Add docs workspace package only after approval.
- Configure Next app-router shell, Fumadocs MDX source, and `lib/source.ts`.
- Create docs route, root layout, docs layout, and root landing page.
- Add `source.config.ts` and content directory.
- Keep package-manager commands as proposed commands until approved.

### W2 — shadcn/ui + Fumadocs theme integration

- Initialize shadcn/ui in the docs app after package-manager approval.
- Configure `components.json`, import aliases, and local UI component path.
- Use the Fumadocs shadcn stylesheet/preset so docs UI follows shadcn tokens.
- Add a small local component set: button, card, badge, alert, tabs, table, accordion, dialog, sheet, tooltip, command.
- Add `/spicetify` domain components for plans, policies, snapshots, audits, and workflow steps.

### W3 — Curated content

- Write curated pages for overview, quickstart, safety, repair, rollback, profile switching, and third-party auditing.
- Add explicit “what `/spicetify` will never do automatically” content.
- Add novice and power-user examples with fake outputs.
- Add docs-site-specific page explaining the documentation architecture.

### W4 — Generated reference pipeline

- Add `scripts/sync-openspec-content.ts` in dry-run-first mode.
- Generate schema reference pages, mode reference pages, requirement pages, and traceability pages into `generated/`.
- Mark generated pages with source paths and regeneration metadata.
- Add tests that generated content never contains secrets or unresolved placeholders.

### W5 — Search, LLM routes, and quality

- Add default local search endpoint and custom search dialog if needed.
- Add `llms.txt` and `llms-full.txt` routes that use redacted, stable docs content.
- Add link checks, content checks, accessibility checks, typecheck, lint, and build scripts.
- Add docs-specific regression prompts and screenshots only if approved.

### W6 — Deployment readiness, not deployment

- Document deployment options and environment variables in `.env.example` only.
- Do not deploy, configure DNS, add analytics, or add hosted search credentials without explicit approval.
- Produce a release-readiness report with validation evidence and unresolved risks.

## Done when

- The docs site has a running local app shell.
- Fumadocs content source and docs routes render curated content.
- shadcn/ui components are local and themed consistently with Fumadocs UI.
- Generated reference pages can be produced deterministically from local sources.
- Search and LLM routes are present or explicitly deferred with reasons.
- Validation passes or failures are documented with owners.
- The site does not weaken `/spicetify` runtime safety language.

## Stop conditions

Stop before proceeding if:

- package installation is needed but not approved,
- a package manager migration is implied,
- Fumadocs or shadcn setup docs conflict with the target repo,
- generated docs would include secrets or private local paths,
- external registries or remote component sources are requested without provenance review,
- deployment, analytics, cloud search, or feedback storage becomes necessary,
- the docs content contradicts OpenSpec requirements.
