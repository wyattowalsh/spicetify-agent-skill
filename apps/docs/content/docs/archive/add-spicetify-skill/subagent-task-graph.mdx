# Subagent task graph for the Codex swarm

**Path:** `apps/docs/content/docs/archive/add-spicetify-skill/subagent-task-graph.mdx`
**Purpose:** Define a swarm-ready, subagent-optimized implementation DAG for `/spicetify` and its Fumadocs + shadcn/ui companion site.
**Status:** Proposed
**Load/use when:** Kicking off implementation with Codex subagents, splitting work across worktrees, or reviewing whether parallel tasks are safe.

## Source posture

Codex subagents are useful here because the work is broad but separable: contracts, command safety, filesystem state, auditing, platform repair, docs-site content, UI, and QA can be explored or implemented in parallel when their write scopes do not overlap. Official Codex docs state that subagents are spawned only when explicitly requested, can run in parallel, and return consolidated results; they also inherit the current sandbox and approval policy. The graph below therefore treats subagents as bounded workers, not autonomous approvers.

## Coordination rules

- The orchestrator owns the task graph, final merge, final report, and any edits to top-level OpenSpec/planning docs after subagents finish.
- A subagent may write only inside its assigned write scope.
- Parallel-safe work requires non-overlapping writes, satisfied dependencies, and independent validation.
- A subagent must stop rather than widen scope, install packages, access network, touch real Spotify/Spicetify state, change permissions, commit, push, deploy, enable hooks/MCP, or request secrets.
- Subagents inherit the caller's sandbox and approval policy; they do not ask to loosen it.
- Subagents must return the standardized result envelope in this document.
- The orchestrator consolidates results before accepting any broad design or API change.
- If conflict appears between agents, the orchestrator pauses implementation and records the decision in `decision-records.md` and `PLANS.md`.

## Swarm topology

```text
orchestrator
  ├─ A0 repository-preflight
  ├─ A1 contracts-and-schemas
  ├─ A2 command-policy-state
  ├─ A3 audit-provenance-security
  ├─ A4 platform-repair-fixtures
  ├─ A5 modes-and-runtime-flows
  ├─ A6 companion-docs-site
  ├─ A7 tests-validation-evals
  └─ A8 final-review-red-team
```

The orchestrator should start with A0. After A0 records repo evidence and write scopes, A1 through A4 can run in parallel. A5 waits for A1/A2/A3/A4 contract outputs. A6 can run in parallel after A0 if it writes only docs-site paths and does not install dependencies. A7 waits for contract paths from A1/A2/A3/A4 and may run in parallel with late A5/A6 implementation checks if write scopes stay separate. A8 runs last.

## Dependency waves

### Wave S0 — Orchestrator preflight

- Assign A0 only.
- Confirm the repository is trusted and writable.
- Confirm the active OpenSpec change path exists.
- Confirm package manager, test commands, current AGENTS guidance, and existing specs.
- Update `context-map.md` and `PLANS.md` with evidence.

### Wave S1 — Parallel foundations

Parallel-safe after A0 completes:

- A1 contracts-and-schemas.
- A2 command-policy-state.
- A3 audit-provenance-security.
- A4 platform-repair-fixtures.
- A6 companion-docs-site in plan-only mode or implementation mode if docs write scope is approved.

A1 through A4 must return interface notes before A5 starts. A6 must not change shared root package files unless explicitly assigned by the orchestrator after A0.

### Wave S2 — Runtime modes

A5 implements modes after A1/A2/A3/A4 provide stable contracts. If A5 discovers a contract gap, it stops and returns a contract-change request rather than patching schemas ad hoc.

### Wave S3 — Validation hardening

A7 adds cross-cutting fake integration tests, evals, validator rules, and docs validation. A7 may run while A5/A6 finish if write scopes do not overlap.

### Wave S4 — Red-team review and consolidation

A8 reviews the full diff, with emphasis on raw shell paths, real Spotify/Spicetify CI escape, missing snapshot before mutation, missing plan-hash confirmation, third-party code trust shortcuts, docs examples that leak secrets or bypass safety, inconsistent OpenSpec/spec/task/design claims, and overlapping subagent write scopes.

The orchestrator applies only review fixes that are within scope and records unresolved issues.

## Worktree strategy

Use separate worktrees or branches only after human approval. Suggested mapping:

| Worktree | Agents | Write scopes |
|---|---|---|
| `contracts` | A1 | schemas and schema tests |
| `runtime-core` | A2, later A5 | core command/policy/state/modes, non-overlapping by phase |
| `security-audit` | A3 | audit/provenance/privacy modules and tests |
| `platform-fixtures` | A4 | platform/recovery/fake fixture paths |
| `docs-site` | A6 | docs app/content/UI paths |
| `validation` | A7 | tests/evals/validator after contract paths stabilize |

Do not let two worktrees independently modify the same OpenSpec spec or root package file without orchestrator consolidation.

## Standard subagent result envelope

Each subagent must return:

```json
{
  "agentId": "A1-contracts-and-schemas",
  "status": "completed | blocked | needs-review",
  "tasksAttempted": ["TASK-010-add-core-schemas"],
  "read": ["schemas/", "api-contracts.md"],
  "writeScopeUsed": ["schemas/", "tests/schemas/"],
  "changedFiles": [],
  "validationsRun": [
    { "command": "python tools/validate_bundle.py --root .", "result": "not-run | passed | failed", "notes": "..." }
  ],
  "findings": [
    { "severity": "medium", "summary": "...", "evidence": "path:line", "recommendedFix": "..." }
  ],
  "handoffNotes": ["..."],
  "stopConditionsHit": []
}
```

## Merge protocol

1. Orchestrator receives all subagent envelopes.
2. Reject any envelope that reports writes outside scope or missing validation notes.
3. Compare changed files for overlap before applying or merging diffs.
4. Resolve schema/API conflicts before runtime implementation continues.
5. Update `PLANS.md` with progress, surprises, decisions, and validation log.
6. Run bundle/project validation.
7. Run A8 final-review-red-team.
8. Produce final Codex report with task IDs, changed files, validation evidence, unresolved risks, and next action.

## Single-agent fallback

If subagents are unavailable, the same graph becomes a sequential implementation plan:

1. A0
2. A1
3. A2
4. A3
5. A4
6. A5
7. A6
8. A7
9. A8

The fallback MUST preserve the same write scopes, stop conditions, validation requirements, and result envelope after each phase.

## Acceptance criteria

- Every parallel task has a non-overlapping write scope.
- Every subagent has bounded reads, writes, output, validation, and stop conditions.
- The kickoff prompt references this graph instead of embedding the entire bundle.
- The orchestrator consolidates all results before any final success claim.
- A single-agent fallback exists and remains safe.
