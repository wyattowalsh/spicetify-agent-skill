# Docs content architecture

**Path:** `apps/docs/content/docs/archive/add-spicetify-skill/docs-content-architecture.mdx`
**Purpose:** Content model, navigation, synchronization, and editorial rules for the accompanying Fumadocs site.
**Status:** Proposed
**Load/use when:** Designing or reviewing the docs information architecture and generated-reference pipeline.

## Content principles

- Progressive disclosure: landing page and quickstart first; deep OpenSpec details behind reference pages.
- Source-grounded: every generated reference page links back to the local OpenSpec/planning/schema source path.
- No giant prompt dumps: docs pages summarize and link; the bundle remains the source of truth.
- Safe examples: all examples use fake paths, fake snapshots, fake plan IDs, and fake audit IDs unless explicitly user-provided and redacted.
- Auditable generation: generated content is deterministic and diffable.

## Navigation model

Use `meta.json` files per section to keep the sidebar curated and avoid exposing raw generated file order. Suggested sidebar order:

1. Overview
2. Quickstart
3. Safety first
4. Core workflows
5. Modes
6. Profiles and manifests
7. Auditing third-party code
8. Repair after Spotify updates
9. Developer scaffolds
10. Reference
11. Implementation guide
12. Docs-site design

## Editorial workflow

1. Update OpenSpec/planning/schema source.
2. Run content sync in dry-run mode.
3. Review generated diff.
4. Run docs validation.
5. Commit generated docs only with the source change that caused them.

## Content validation rules

- No unresolved `pending` or placeholder braces in user-facing docs.
- No release labels or numbered artifact lineage.
- No raw secrets, tokens, cookies, private user paths, or real Spotify prefs content.
- No instructions that bypass `/spicetify` confirmation gates.
- No external registry install instructions without explicit approval note.
- All internal links resolve.
- All code blocks either compile, are marked as pseudocode, or are marked as proposed shell commands.
- Generated pages include source-path metadata.

## LLM docs model

The site should expose AI-readable routes, but only after applying the same redaction rules used for reports. The generated text should prioritize stable content:

- mode summaries,
- safety invariants,
- schema summaries,
- command registry boundaries,
- repair flow summaries,
- eval stack and evolve summaries,
- Codex handoff read order.

It should omit local paths, user reports, screenshots, DevTools logs, and any private run records.
