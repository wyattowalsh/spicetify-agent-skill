# Audit review: `/spicetify` planning bundle

**Path:** `apps/docs/content/docs/archive/add-spicetify-skill/audit-review.mdx`
**Purpose:** End-to-end critique and repair log for the planning bundle.
**Status:** Proposed
**Load/use when:** Reviewing why the bundle is structured as it is, or deciding what to implement first.

## Verdict

The plan is strong and buildable, but a safe local /spicetify skill must treat command semantics, platform packaging, third-party code, DevTools logs, launch flags, and update repair as first-class safety domains. The refined bundle keeps the skill name simple (`/spicetify`) and strengthens the implementation handoff without adding release labels or alternate product names.

## Design critique

The best architecture is not a universal command wrapper. It is a policy-enforced desired-state planner that can compile a small set of semantic operations into allowed Spicetify commands and file transactions. This keeps the user experience powerful while preventing the highest-risk failure modes: arbitrary shell execution, accidental user-file overwrites, third-party JavaScript enablement, install-script execution, path traversal, and failed repair loops after Spotify updates.

## Recommended implementation posture

1. Build the safety kernel first: schemas, command registry, policy engine, fake Spicetify, path guard, snapshot store.
2. Keep all mutating mode implementations behind the same `OperationPlan` → policy → snapshot → execute → verify → report state machine.
3. Defer network/GitHub/Marketplace installation until provenance locking, archive extraction tests, and third-party audit fixtures pass.
4. Treat DevTools, launch flags, log collection, screenshots, and package/permission changes as opt-in power-user features.
5. Do not let a successful CLI exit code alone update last-known-good; verification must pass.

## Regression additions required

- Ambiguous “update Spicetify” prompts produce a disambiguation plan, not immediate execution.
- A Marketplace install script in docs is refused as automation and converted to a manual/audit plan.
- Snap Spotify fixture blocks mutation.
- Nix/Home Manager fixture recommends export/declarative mode.
- Remote debugging flags require separate confirmation and rollback.
- Existing plan approval is invalidated by config/assets/source-hash drift.
- Third-party code with token access plus network send is blocked.

## Remaining implementation uncertainties

- QUESTION-001: Which exact target repository conventions, package manager, and test runner will the coding agent inherit?
- QUESTION-002: Will the implementation ship as a CLI, a Codex Skill, or both from the start?
- QUESTION-003: Should the first shipped profile format support declarative Nix/Home Manager export, or only local JSON/YAML manifests?
- QUESTION-004: Should Marketplace-aware install assistance remain manual-only until local file and GitHub archive flows are proven?
