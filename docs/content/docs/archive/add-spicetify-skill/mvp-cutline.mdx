# MVP cutline

**Path:** `docs/content/docs/archive/add-spicetify-skill/mvp-cutline.mdx`
**Purpose:** Recommended smallest useful implementation slice and deferred capabilities.
**Status:** Proposed
**Load/use when:** Deciding task sequencing or limiting scope for the first build.

## MVP principle

Ship the smallest `/spicetify` build that can safely mutate local state. Do not ship remote installs, custom package scripts, DevTools log collection, or screenshot automation until the safety kernel is proven.

## Must ship first

| Capability | Why it is in MVP | Exit criteria |
|---|---|---|
| Schemas and JSON validation | Every plan/report/audit/snapshot must be machine-checkable. | All schemas parse; valid/invalid fixtures exist. |
| Command registry and runner | Safety depends on allowlisted argv with `shell:false`. | Raw shell and shell metacharacter tests fail closed. |
| Fake Spicetify environment | CI must never mutate real Spotify. | Integration tests run against fake binary and temp roots only. |
| Path guard and staged transactions | Prevent overwrite/path traversal/symlink escape. | Traversal and symlink fixtures are rejected. |
| Policy engine | Risk/approval gates must precede mutation. | Policy decision emitted for every plan. |
| Snapshot store | Rollback must exist before mutation. | Snapshot/restore hash tests pass. |
| Inspect and doctor | Read-only grounding is required for all repair/mutation. | Platform and broken-state fixtures produce expected findings. |
| Config/profile local flows | Profiles are core power-user value and lower risk than remote code. | Exact desired-state diff and no-op repeat tests pass. |
| Theme create/switch local flows | Useful visible feature with controllable file writes. | Generated theme validates and rolls back in fake env. |
| Apply and rollback | Transactional value proposition. | Failed apply restores previous config/assets. |
| Repair after Spotify update | Top user story; must be safe and honest. | Backup/apply path works; destructive fallback is separate confirmation. |
| Report/redaction | Users need shareable diagnostics. | Reports validate and redact token-like content. |

## Deferred capabilities

| Deferred capability | Why deferred | Unlock condition |
|---|---|---|
| GitHub/theme archive installation | Requires network approval, pinning, archive traversal tests, provenance lock, and malicious fixture coverage. | Provenance and archive-extraction tests pass. |
| Generated extension/custom-app build execution | Requires package-manager and lifecycle-script policy plus template build isolation. | Local generated-template tests pass without third-party scripts. |
| Marketplace install automation | Marketplace install scripts and in-app state are higher-risk than local file plans. | Manual inspect/audit/report path is reliable and documented. |
| DevTools log collection | Logs can leak tokens/PII and remote debugging expands attack surface. | Consent, redaction, local-only debugging, and rollback tests pass. |
| Screenshot verification | Platform/UI automation is flaky and privacy-sensitive. | Optional screenshot provider has consent, redaction, and fixture coverage. |
| Full uninstall/cleanup | Deletions are irreversible without excellent ownership tracking. | Skill-owned file tracking and pre-delete snapshot tests pass. |

## Build order

1. Contracts: schemas, error taxonomy, command registry.
2. Safety kernel: path guard, policy, snapshots, provenance skeleton, redaction.
3. Read-only modes: inspect, doctor, audit, report.
4. Local mutation modes: config, profile, theme, apply, rollback.
5. Repair mode: doctor → snapshot → backup/apply → verify, fallback as separate plan.
6. Deferred power-user modes only after safety fixtures cover the risk.

## Definition of Done for MVP

- No arbitrary shell execution path exists.
- All command execution uses `spawn("spicetify", args, { shell: false })` or equivalent.
- Every mutating plan has a policy decision, snapshot requirement, verification steps, rollback path, and plan hash.
- Fake integration tests pass without real Spotify or real Spicetify.
- Last-known-good is not updated unless verification succeeds.
- Reports are redacted and schema-valid.

## Docs-site cutline

Include in the practical build sequence after runtime safety kernel and read-only modes:

- docs app shell,
- Fumadocs content source,
- shadcn theme integration,
- overview/quickstart/safety/mode catalog,
- generated schema and requirement references,
- local search,
- AI-readable routes,
- content validation.

Defer deployment, analytics, hosted search credentials, feedback backends, external registries, and public publishing until explicitly approved.
