# Docs site design system

**Path:** `docs/content/docs/archive/add-spicetify-skill/docs-site-design-system.mdx`
**Purpose:** shadcn/ui customization plan for the companion Fumadocs site.
**Status:** Proposed
**Load/use when:** Designing, reviewing, or implementing docs-site UI components.

## Visual direction

The site should feel like a precise terminal-grade operations manual: dark by default, crisp typography, readable code blocks, restrained motion, and safety states that are impossible to miss. It should not imitate Spotify branding or imply endorsement.

## shadcn/ui setup assumptions

- Use the target repo's package manager and workspace conventions.
- Use TypeScript and React Server Components unless the target repo requires otherwise.
- Keep `components.json` in the docs workspace when using a monorepo.
- Prefer CSS variables for theme tokens so risk states, code blocks, and dark mode remain consistent.
- Add components only when a page requires them.

| `textarea` / `input` | shadcn/ui | Use only for static examples or client-side demos that do not touch local state. |

## Custom variants

| Variant | Meaning |
|---|---|
| `risk-read` | Read-only, no mutation. |
| `risk-low` | Low local generated-file risk. |
| `risk-medium` | Mutating but bounded and snapshot-protected. |
| `risk-high` | Explicit confirmation required. |
| `risk-blocked` | Refused or manual-only action. |
| `source-verified` | Official or local source fact. |
| `source-assumed` | Labeled assumption. |
| `source-untrusted` | Third-party data requiring audit. |

## Custom components


### `RiskBadge`

Renders risk states (`read`, `low`, `medium`, `high`, `blocked`) with both color and text so safety status is accessible and unambiguous.

### `ModeCard`

Renders a mode's purpose, risk, preconditions, plan shape, and rollback support.

### `OperationTimeline`

Shows dry-run, policy decision, snapshot, confirmation, execution, verification, and rollback. This should be used on all pages that discuss mutation.

### `CommandPlan`

Displays allowlisted argv arrays as structured data. It must never render arbitrary shell snippets as a recommended command.

### `SchemaViewer`

Summarizes JSON schemas with required fields, enum values, and examples. It should link to raw schema files in the repo.

### `RecoveryCallout`

Maps error codes to safe next actions and stop rules.

### `ProvenancePanel`

Shows source type, ref, hash, audit verdict, and whether an approval can be reused.

## Accessibility requirements

- All color-coded risk states also include text labels.
- All interactive examples are keyboard reachable.
- Code blocks have copy buttons only for safe, proposed commands and must label commands as review-required when mutating.
- Motion is optional and respects reduced-motion preferences.
- Tables must remain readable on narrow screens.

## Copy rules

- Never say `/spicetify` will “just fix” Spotify; say it will inspect, plan, repair if supported, and stop when unsupported.
- Never present shell as casual copy-paste for mutating actions; label commands as proposed validation or target-repo setup.
- Use “requires confirmation” and “manual-only” consistently.
- Mark third-party code as untrusted until audited.
