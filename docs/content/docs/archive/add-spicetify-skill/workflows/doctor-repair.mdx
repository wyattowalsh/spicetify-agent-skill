# Workflow: doctor and repair

**Path:** `docs/content/docs/archive/add-spicetify-skill/workflows/doctor-repair.mdx`
**Purpose:** Diagnose and repair broken Spicetify state, especially after Spotify updates.
**Status:** Proposed
**Load/use when:** Implementing `doctor`, `repair`, `update`, or platform repair fixtures.

## Doctor sequence

1. Discover Spicetify binary, CLI identity string, config path, userdata path, and Spotify path.
2. Parse `config-xpui.ini`.
3. Check platform/package hints from paths and config.
4. Check current theme folder, required theme files, color scheme, optional `theme.js` policy.
5. Check enabled extensions exist, are leaf `.js` files, parse syntactically, and match provenance when known.
6. Check custom apps exist and have valid `manifest.json` and `index.js`.
7. Check Marketplace presence if enabled or requested.
8. Check risky launch flags without printing sensitive values.
9. Optionally collect redacted DevTools logs only with consent.
10. Produce findings with severity, evidence, safe repair candidates, and stop conditions.

## Post-Spotify-update repair

Primary safe plan:

```text
doctor -> snapshot -> spicetify backup -> verify backup evidence -> spicetify apply -> verify -> report
```

Fallback plan, separate high-risk confirmation only:

```text
spicetify restore -> spicetify backup -> spicetify apply -> verify
```

The fallback is never hidden behind a generic “fix it” confirmation because it can alter patched Spotify state.

## Update ambiguity handling

When the user says “update,” doctor/repair must classify the request:

- Spotify updated and Spicetify broke → repair flow.
- Theme CSS changed and needs reload → hot-reload plan using `spicetify update`.
- Spicetify itself may need maintenance → install-method-specific plan; no package managers or install scripts without approval.
- User pasted docs/README update command → no execution until intent and risk are explicit.

## Repair boundaries

Repair MAY:

- re-run backup/apply after snapshot;
- fix broken config references by disabling missing assets after confirmation;
- regenerate skill-owned theme/profile files;
- produce manual package/path/permission instructions;
- preserve a debug state if the user explicitly asks not to rollback after failure.

Repair MUST NOT automatically:

- install or update Spotify/Spicetify/package managers;
- run scripts from docs, READMEs, package metadata, or Marketplace;
- modify permissions or app bundles;
- edit launch flags without explicit DevTools/update plan;
- delete user assets without a restore point;
- suppress unsupported-compatibility warnings or loop repair commands.

## Platform stop examples

| Finding | Repair behavior |
|---|---|
| Snap Spotify | Block mutation and report manual package switch options. |
| AUR/APT permission issue | Report needed path/permission action; do not run `sudo` or `chmod`. |
| Flatpak path/prefs mismatch | Propose exact config changes after full path discovery. |
| Microsoft Store Spotify prefs issue | Diagnose and suggest manual migration path. |
| Declarative Nix/Home Manager ownership | Prefer export/manifest mode; warn local edits may be overwritten. |

## Verification after repair

- CLI exit codes and stderr classification.
- config parse/diff.
- required files exist and match planned hashes when relevant.
- doctor has no new high findings.
- optional screenshot/log checks are consent-gated and redacted.
- last-known-good update only after success.

## Regression tests

- post-update fixture succeeds via backup/apply.
- restore fallback is present but not executed without separate confirmation.
- unsupported compatibility output stops.
- platform permission issue emits manual report only.
- ambiguous “update” prompt produces disambiguation.
