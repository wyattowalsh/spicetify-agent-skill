# Project brief: `/spicetify`

**Path:** `docs/content/docs/archive/add-spicetify-skill/index.mdx`
**Purpose:** Read-first planning brief for humans and coding agents.
**Status:** Proposed
**Load/use when:** Orienting to the planning pack after root `README.md`.

## Objective

Create `/spicetify`, a compact AI skill plus installable local Python engine that safely orchestrates Spicetify CLI and local Spicetify customization workflows through semantic modes, dry-run plans, snapshots, audits, verification, and rollback.

## Primary actors

- Novice user who wants safe customization and repair.
- Power user who wants profile manifests, repeatability, and automation.
- Extension/theme/custom-app developer who wants scaffolding, watch/debug loops, and audit.
- Coding agent implementing the plan.

## Scope

In scope:

- all 23 semantic modes listed in `mode-catalog.md`
- OpenSpec requirements and Codex handoff
- command registry, policy, snapshots, rollback, audit, verification
- fake Spicetify environments and schema-validated structured output

Out of scope:

- real Spotify/Spicetify mutation by default or in CI
- install scripts, package-manager actions, permission changes, publishing, pushes, PRs, deploys
- bypassing Spotify account restrictions, ads, DRM, or terms

## Assumptions

- ASSUMPTION-001: The implementation runs locally with user-level permissions.
- ASSUMPTION-002: Python is the runtime authority for the installable `spicetify-agent` CLI; TypeScript/Node.js is isolated to the companion docs app and generated customization templates.
- ASSUMPTION-003: Spicetify and Spotify paths must be discovered from the user's machine at runtime.
- ASSUMPTION-004: The implementation root is this repository; future coding still starts with read-only repo grounding.
